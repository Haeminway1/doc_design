#### 문제 2: CSS가 파일마다 중복·불일치

```
파일별 CSS 불일치 사항:
├── 폰트 로딩: @import vs <link>, Playfair/Merriweather/Inter 혼재
├── body line-height: 1.4 / 1.6 / 1.65 / 1.7 / 1.8 (5가지)
├── 페이지 padding: 12mm~20mm (4가지 이상)
├── 폰트 크기 단위: pt / px 혼용
├── 푸터 구현: absolute vs flex / CSS counter vs 하드코딩 번호
├── `:root` 중복 선언: 일부 파일에서 같은 파일 내 2~3회 재선언
└── 인라인 스타일 남발: style="font-size: 12.5px;" 등 직접 삽입
```

- 교재 간 문제 공유/이동 시 HTML+CSS를 통째로 복붙해야 함

## 2. 목표 아키텍처 제안

### 2.0 핵심 문제: "규격 vs 유연성" 딜레마

모든 콘텐츠를 JSON으로 넣으면 규격은 잡히지만 디자인적 매력이 죽는다. 실제 교재를 분석하면 콘텐츠는 두 가지 성격으로 명확히 나뉜다:

```
┌─────────────────────────────────────────────────────────┐
│                    교재 콘텐츠                            │
├──────────────────────┬──────────────────────────────────┤
│  🔷 정형 데이터       │  🎨 자유 콘텐츠                   │
│  (항상 같은 구조)      │  (페이지마다 다른 구성)            │
├──────────────────────┼──────────────────────────────────┤
│  - 객관식 문제        │  - 개념 해설 (비교표+공식+서사)     │
│  - 선택지 5개         │  - Vera's Tip 박스 (자유 구성)     │
│  - 정답 번호          │  - 문법 도표 (커스텀 스타일)        │
│  - 단어 엔트리        │  - 구문 분석 (색상 마킹+화살표)     │
│  - 정답 그리드        │  - 단원 도입 페이지                │
│                      │  - 비교/대조 설명                  │
│                      │  - 인라인 예제와 해설 흐름          │
├──────────────────────┼──────────────────────────────────┤
│  → JSON              │  → HTML (디자인 시스템 컴포넌트)    │
│  (자동 생성 가능)      │  (직접 작성, 자유롭게 조합)         │
└──────────────────────┴──────────────────────────────────┘
```

**결론: 하이브리드 아키텍처**
- **정형 데이터** → JSON으로 관리, 템플릿으로 자동 렌더링
- **자유 콘텐츠** → HTML로 직접 작성하되, 공유 디자인 시스템의 컴포넌트 클래스를 조립
- **디자인 시스템** → 두 세계 모두를 관통하는 통일된 CSS 레이어

이렇게 하면:
- 문제 200개 수정 → JSON만 고치면 끝 (규격)
- 가정법 해설에 비교표+팁박스+공식 → HTML에서 자유롭게 배치 (유연성)
- 두 영역 모두 같은 색상/폰트/간격 토큰을 쓰므로 시각적 통일감 유지 (일관성)

### 2.1.1 템플릿 시스템 상세

**"테마 = 색상만 바꾸기"가 아니다.** 템플릿은 교재의 전체적인 무드와 스타일을 결정하는 독립적인 디자인 세트다.

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         시스템 구조 (레이어)                              │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  Layer 0: base/       공통 기반 (A4, 리셋, 인쇄)     ← 절대 불변        │
│     ↓                                                                   │
│  Layer 1: components/ 컴포넌트 구조 + 기본 스타일     ← 구조적 규격      │
│     ↓                                                                   │
│  Layer 2: templates/  시각적 개성 (색상, 폰트, 표지)  ← 자유 확장        │
│                                                                         │
│  ┌───────────┐  ┌───────────┐  ┌───────────┐  ┌───────────┐           │
│  │ocean-blue │  │royal-purple│ │ earth-tone │  │ mint-sky  │  ← ...    │
│  │ 문법 Bridge│  │ 독해 Basic │ │ 독해 Bridge│  │ 구문독해  │           │
│  │ 문법 Adv  │  │           │ │            │  │           │           │
│  └───────────┘  └───────────┘  └───────────┘  └───────────┘           │
│   교재 A,B가      교재 C가       교재 D가        교재 E가              │
│   공유 가능        단독 사용      단독 사용       단독 사용              │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**템플릿 매니페스트 (`_manifest.json`):**

```jsonc
// system/templates/ocean-blue/_manifest.json
{
  "id": "ocean-blue",
  "name": "Ocean Blue",
  "description": "바다색 기반, 따뜻한 베이지 포인트. 문법 교재 시리즈에 적합.",
  "version": "1.0.0",
  "author": "Vera's Flavor",
  "preview": "preview.png",
  "fonts": {
    "heading": "Inter",
    "body": "Noto Sans KR",
    "accent": "Playfair Display"
  },
  "palette": {
    "primary": "#345C85",
    "primaryLight": "#6C96C6",
    "secondary": "#F2C994",
    "accent": "#E9A266"
  },
  "coverStyle": "border-frame",     // border-frame | full-bleed | split-layout | minimal
  "borderRadius": "0px",            // 0px = 샤프, 8px = 라운드
  "headerStyle": "underline",       // underline | filled | minimal
  "usedBy": ["grammar-bridge-*", "grammar-advanced"]
}
```

**템플릿 간 차이점 예시:**

| 속성 | ocean-blue (문법) | royal-purple (독해 Basic) | earth-tone (독해 Bridge) |
|------|-------------------|--------------------------|--------------------------|
| 메인 색상 | `#345C85` 딥블루 | `#9B59B6` 보라 | `#4B5358` 차콜 |
| 영문 폰트 | Inter + Playfair | Inter | Merriweather |
| 표지 스타일 | 테두리 프레임 | 오버레이 + 중앙 정렬 | 미니멀 보더 |
| 모서리 | 0px (샤프) | 8px (라운드) | 0px (샤프) |
| 헤더 | 밑줄 구분선 | 컬러 밑줄 | 점선 |
| 표 스타일 | 상단 굵은 선 | 전체 테두리 | 최소 라인 |
| 팁박스 | 연한 블루 배경 | 연한 보라 배경 | 베이지 배경 |

**새 템플릿 만들기:**
1. `_template-starter/` 복사
2. `tokens.css`에서 색상/폰트 변수 채우기
3. `cover.css`에서 표지 레이아웃 커스텀
4. `components.css`에서 원하는 컴포넌트만 오버라이드
5. `_manifest.json` 작성
→ 기존 교재에 영향 없이 새 스타일 추가 완료

**교재 매니페스트에서 템플릿 지정:**

```yaml
# books/grammar-bridge-ch02.yaml
book:
  title: "편입영어 문법 Bridge 편 Part 2"
  template: ocean-blue     # ← 이 한 줄로 전체 시각 스타일 결정
  # template: royal-purple  ← 이걸로 바꾸면 같은 내용, 다른 디자인
```

### 2.2 교재 조립 매니페스트 (YAML)

각 교재는 **"어떤 콘텐츠를 어떤 순서로 조립할 것인가"**를 선언하는 매니페스트로 관리된다.
자유 콘텐츠(HTML)와 정형 데이터(JSON)를 자유롭게 섞어 배치할 수 있다.

```yaml
# books/grammar-bridge-ch02.yaml
book:
  title: "편입영어 문법 Bridge 편 Part 2"
  subject: grammar
  level: bridge
  theme: grammar          # → system/themes/grammar.css 적용

pages:
  # 표지 — 자유 콘텐츠
  - type: content
    src: content/grammar/bridge/ch02-cover.html

  # 목차 — 자동 생성
  - type: toc
    auto: true

  # 개념 해설 — 자유 콘텐츠 (비교표, 팁박스, 공식 등 자유 배치)
  - type: content
    src: content/grammar/bridge/ch02-concepts.html

  # Vera's Tip — 자유 콘텐츠
  - type: content
    src: content/grammar/bridge/ch02-tips.html

  # 심화 도표 — 자유 콘텐츠
  - type: content
    src: content/grammar/bridge/ch02-advanced-tables.html

  # 연습문제 — JSON 자동 렌더링 (51~100번)
  - type: problems
    src: data/grammar/bridge/ch02-problems.json
    range: [51, 100]
    layout: two-per-page     # 페이지당 2문제

  # 정답 그리드 — JSON 자동 렌더링
  - type: answer-grid
    src: data/grammar/bridge/ch02-problems.json
    range: [51, 100]

  # 정답 해설 — JSON 자동 렌더링
  - type: explanations
    src: data/grammar/bridge/ch02-problems.json
    range: [51, 100]
    layout: compact          # 페이지당 5개
```

**이 방식의 핵심:**
- 개념 해설 페이지에서는 HTML로 비교표, 팁박스, 공식을 **자유롭게** 배치
- 문제/정답/해설은 JSON에서 **자동으로** 렌더링
- 같은 교재 안에서 두 방식이 **자연스럽게 공존**
- 교재 구성(순서, 포함 범위)을 YAML 한 파일에서 한눈에 파악 가능

### 2.3 자유 콘텐츠의 작성 방식: "레고 블록 조립"

자유 콘텐츠 HTML은 **완전한 백지가 아니다.** 디자인 시스템이 제공하는 컴포넌트 클래스를 "레고 블록"처럼 조립한다. 규격은 지키되, 조합은 자유.

```html
<!-- content/grammar/bridge/ch02-concepts.html -->
<!-- 이 파일은 vera-design-system.css 위에서 동작한다 -->
<!-- 어떤 컴포넌트를 어떤 순서로 배치할지는 작성자 자유 -->

<div class="page">
  <div class="page-header">
    <span class="eng-text">Vera's Flavor - Bridge</span>
    <span>Part 2: 가정법</span>
  </div>
  <div class="page-content">

    <!-- 🧱 섹션 제목 컴포넌트 -->
    <h2 class="section-title">가정법의 심층 이해: 시제 이동과 심리적 거리감</h2>

    <!-- 📝 자유 서술 (디자인 시스템의 본문 스타일 자동 적용) -->
    <p><b>가정법(Subjunctive Mood)</b>은 화자가 특정 상황을 '비현실'의 영역으로
    설정하고 있음을 보여주는 문법적 장치입니다...</p>

    <!-- 🧱 비교표 컴포넌트 (디자인 시스템 제공) -->
    <table class="comparison-table">
      <thead><tr><th>구분</th><th>직설법</th><th>가정법</th></tr></thead>
      <tbody>
        <tr>
          <td><b>현재</b></td>
          <td>the data <b>is</b> insufficient</td>
          <td>If the data <b>were</b> sufficient</td>
        </tr>
        <!-- 내용은 자유, 스타일은 통일 -->
      </tbody>
    </table>

    <!-- 🧱 공식 박스 컴포넌트 -->
    <div class="rule-box">
      <p>If + S + <b>과거동사</b>, S + would/could + <b>R</b></p>
    </div>

  </div>
  <div class="page-footer"></div>
</div>

<div class="page">
  <div class="page-header">...</div>
  <div class="page-content">

    <!-- 🧱 팁 박스 컴포넌트 (안에 뭘 넣을지는 자유) -->
    <div class="tip-box">
      <div class="tip-box-title">Vera's Flavor Tip</div>

      <h4 class="tip-subtitle">1. 가정법 시제 = 현실과의 '거리'</h4>
      <p>가정법 시제를 공식으로만 외우면 혼합가정법에서 무너집니다...</p>

      <!-- 안에 리스트를 넣든, 표를 넣든, 예문을 넣든 자유 -->
      <ul class="styled-list">
        <li><b>거리 0:</b> He <b>is</b> not a diligent student.</li>
        <li><b>거리 1:</b> I wish he <b>were</b>... (가정법 과거)</li>
        <li><b>거리 2:</b> If he <b>had been</b>... (가정법 과거완료)</li>
      </ul>

      <!-- 코드 블록 예시도 자유롭게 -->
      <div class="example-highlight">
        If the scientist <b>had not made</b> that error <i>in his previous experiment</i>,
        his theory <b>would be</b> accepted <i>today</i>.
      </div>
    </div>

    <!-- 🧱 주의 박스 컴포넌트 -->
    <div class="warning-box">
      <h4>If절에 would를 쓰면 안 되는 이유</h4>
      <p>조건절은 '상상의 전제'이므로 과거 시제만 허용됩니다...</p>
    </div>

  </div>
  <div class="page-footer"></div>
</div>
```

**핵심 비유: "레고 블록"**
- 블록의 색상, 크기, 모양 = **디자인 시스템이 결정** (통일)
- 블록을 어떤 순서로 쌓을지 = **작성자가 결정** (자유)
- 같은 블록으로 탑을 만들 수도, 집을 만들 수도 있음

### 2.4 핵심 설계 원칙 (수정)

| 원칙 | 설명 |
|------|------|
| **하이브리드 콘텐츠 관리** | 정형 데이터(문제/어휘)는 JSON, 자유 콘텐츠(해설/팁)는 HTML로 이원화 |
| **단일 디자인 시스템** | 두 세계 모두 `vera-design-system.css` 하나를 공유 |
| **컴포넌트 = 레고 블록** | 자유 콘텐츠는 백지가 아니라 디자인 시스템 블록의 자유 조합 |
| **테마 = 색상 오버라이드만** | 과목별 차이는 CSS 변수 몇 개만 덮어씌움 |
| **매니페스트 기반 조립** | 교재 구성을 YAML로 선언, 빌드 시 자동 조립 |

## 3. JSON Schema 정의 (정형 데이터)

> JSON에는 **반복 구조를 가진 정형 데이터만** 들어간다. 개념 해설, 팁, 도표 같은 자유 콘텐츠는 HTML로 관리 (섹션 2.3 참조).
> 모든 JSON 파일은 `schemas/` 디렉토리의 JSON Schema로 **빌드 시 자동 검증**된다.

### 3.0 네이밍 컨벤션 (전체 적용)

```
■ ID 네이밍 규칙:
  {subject}-{level}-{unit}-{sequence}
  예: grammar-bridge-ch02-051
      reading-basic-p01-q01
      voca-basic-d01-003

■ 파일명 규칙:
  {unit}-{contentType}.json
  예: ch02-problems.json
      day-01-words.json
      part01-passages.json

■ 필드명 규칙:
  - camelCase 사용 (snake_case 금지)
  - 약어 금지: explanation (O), exp (X) / instruction (O), inst (X)
  - boolean은 is/has 접두어: isRequired, hasImage

■ enum 값 규칙:
  - kebab-case 사용
  - 예: "fill-in-blank", "error-find", "main-idea"
```

### 3.5 스키마 검증 파이프라인

```
[빌드 시 자동 실행]

1. node build/validate-schema.js
   ├── data/**/*.json 전체 스캔
   ├── 각 파일의 $schema 필드 읽기
   ├── schemas/ 디렉토리에서 해당 스키마 로드
   ├── ajv 라이브러리로 검증
   └── 실패 시 에러 리포트 + 빌드 중단

2. 검증 항목:
   ├── 필수 필드 누락 체크
   ├── ID 패턴 검증 (정규식)
   ├── enum 값 검증 (오타 방지)
   ├── answer 범위 검증 (choices 길이 이내)
   ├── ID 중복 검증 (전체 파일 간)
   └── 참조 무결성 (매니페스트의 src 파일 존재 여부)
```

### 3.6 "이건 JSON? 이건 HTML?" 판별 기준

| 질문 | Yes → JSON | No → HTML |
|------|-----------|-----------|
| 같은 구조가 10회 이상 반복되는가? | 문제 200개, 단어 700개 | 개념 해설, 팁 |
| 필드 3~5개로 완전히 표현 가능한가? | 문제(stem+choices+answer) | 비교표+공식+서사 혼합 |
| 순서만 바꿔도 의미가 유지되는가? | 문제 순서, 단어 순서 | 해설의 논리적 흐름 |
| 자동 렌더링해도 품질이 유지되는가? | 선택지 5개 나열 | 페이지별 커스텀 레이아웃 |

```
실제 적용 예시:

문법 교재 (ch02-가정법):
  ├── ch02-concepts.html    ← HTML (비교표 + 시제이동 설명 + 공식)
  ├── ch02-tips.html        ← HTML (Vera's Tip 3개, 각각 다른 구성)
  ├── ch02-tables.html      ← HTML (심화 도표, 커스텀 스타일)
  └── ch02-problems.json    ← JSON (51~100번 문제+정답+해설, 스키마 검증됨)

독해 교재 (basic):
  ├── part-intro.html       ← HTML (파트 도입 페이지)
  └── passages.json         ← JSON (지문 40개 + 어휘 + 문제 전체, 스키마 검증됨)

보카 교재:
  ├── vera-word-intro.html  ← HTML (Vera's 단어 학습법 소개 페이지)
  └── day-01~70.json        ← JSON (단어 700개 + 테스트 전체, 스키마 검증됨)
```

### 3.7 해설 데이터 관리

해설에는 두 종류가 있다:

| 구분 | 예시 | 관리 방식 |
|------|------|----------|
| **문제별 짧은 해설** | "가정법 과거완료. If절은 had p.p., 주절은 would have p.p." | JSON `explanation` 필드 (마크다운 허용) |
| **개념 심화 해설** | 비교표+공식+예문분석이 2페이지에 걸쳐 전개 | HTML 자유 콘텐츠 |

JSON 내 짧은 해설에서 허용되는 마크다운 문법:

```
허용:  **볼드**  `코드`  - 리스트  \n (줄바꿈)
금지:  # 헤더  | 테이블 |  ![이미지]()  <html태그>
```

   a. system/components/에 CSS 추가
   b. 사용 가이드 .md 작성
   c. _index.md에 등록
4. 기존 컴포넌트 개선 시
   a. 버전 코멘트 남기기 (/* v2.0 - 간격 조정 */)

## 9. 부록: 왜 "전부 JSON"이 아닌 하이브리드인가

```
❌ 전부 JSON:
   {
     "concept": "가정법은 시제를 후퇴시켜서...",  ← 비교표는? 공식박스는? 팁은?
     "table": [["구분","직설법","가정법"],...],     ← 셀 병합은? 볼드는? 색상은?
     "tip": "시제 = 거리 개념으로..."              ← 리스트는? 코드블록은? 예문하이라이트는?
   }
   → 결국 HTML을 문자열로 넣게 됨 = "JSON 안의 HTML" = 최악의 조합

❌ 전부 HTML (현재):
   → 문제 200개 × 선택지 5개 × 해설 = 수천 줄의 반복 마크업
   → 오타 하나 고치려면 HTML 태그 사이를 헤매야 함
   → CSS가 파일마다 중복

✅ 하이브리드:
   반복 데이터(문제/단어) → JSON  (관리 효율)
   서사 콘텐츠(해설/팁)  → HTML  (표현 자유)
   디자인(색상/폰트/간격) → CSS   (통일 규격)
   조립 순서             → YAML  (한눈에 파악)
```

이 구조에서 새 교재를 만드는 과정:

```
1. YAML 매니페스트 작성 (5분)
   → "표지, 도입, 개념해설, 문제 1~50, 정답, 해설" 순서 선언

2. JSON에 문제 입력 (반복 작업, 도구 지원 가능)
   → 문제/선택지/정답/짧은해설

3. HTML로 개념 해설 작성 (창의적 작업)
   → 디자인 시스템 컴포넌트를 자유 조합
   → .concept-box, .tip-box, .comparison-table 등 레고 블록 사용

4. node build/assemble.js --book 새교재
   → 자동 조립 + PDF 생성
```
