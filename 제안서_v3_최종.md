# 1. 제안 개요

## 1.1 본 문서의 목적

본 제안서는 HLS 학원의 학생 관리 업무를 **수기 스프레드시트 방식**에서 **자동화된 디지털 시스템**으로 전환하기 위한 개발 계획과 비용을 제시합니다.

## 1.2 핵심 제안 요약

```
┌───────────────────────────────────────────────────────────────────────┐
│                         제안 구조 한눈에 보기                            │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│     Part 1 (지금 바로 시작)              Part 2 (검증 후 확장)           │
│     ┌─────────────────────┐           ┌─────────────────────┐        │
│     │  스마트 학원 관리     │    ──→    │  통합 디지털 시스템   │        │
│     │  (GAS 프로토타입)     │           │  (웹 + 모바일 앱)    │        │
│     │                     │           │                     │        │
│     │  • 학생/반/출결 관리  │           │  • 학생관리 SaaS     │        │
│     │  • 상담 기록         │           │  • 문제풀이 앱       │        │
│     │  • 자동 문자 발송     │           │  • 문제 DB 관리      │        │
│     │                     │           │                     │        │
│     │  400만원 (일시불)     │           │  월 700~800만원      │        │
│     │  운영비: 0원         │           │  (옵션에 따라)       │        │
│     │  완성: 약 1개월       │           │  개발: 10~12개월     │        │
│     └─────────────────────┘           └─────────────────────┘        │
│                                                                       │
│     * Part 2와 함께 계약 시                                            │
│       Part 1: 200만원 (50% 할인)                                      │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

| 구분 | Part 1 | Part 2 |
|------|--------|--------|
| **명칭** | 스마트 학원 관리 (프로토타입) | 통합 디지털 학원 시스템 |
| **범위** | 학생/반/출결/상담 관리 | 학생관리 + 문제풀이앱 + 문제DB |
| **플랫폼** | 웹 (Google Apps Script) | 웹 + 모바일 앱 (풀스택) |
| **대상 학원** | HLS 본원 전용 | 다지점 확장 가능 |
| **개발 비용** | 400만원 (일시불) | 월 700~800만원 (옵션별) |
| **운영 비용** | 무료 | 월 5~10만원 |
| **개발 기간** | 약 1개월 | 10~12개월 |

## 1.3 왜 2단계로 진행하는가

대규모 시스템을 처음부터 개발할 경우 다음과 같은 위험이 있습니다:

| 위험 요소 | 설명 |
|----------|------|
| 개발 기간 장기화 | 실제 사용까지 1년 이상 소요 가능 |
| 요구사항 불일치 | 완성 후 실제 업무 흐름과 맞지 않을 수 있음 |
| 비용 집중 | 초기에 대규모 투자 필요 |
| 검증 불가 | 중간 결과물 없이 최종 완성까지 확인 불가 |

**2단계 접근의 장점**:
- Part 1으로 **즉시 업무 효율화** 시작
- 실사용을 통한 **요구사항 검증** 및 보완
- Part 2 진행 여부를 **데이터 기반으로 판단** 가능
- **비용 분산**으로 재정 부담 완화

```
[현재]                [Part 1]                [Part 2]
스프레드시트    →    웹 기반 시스템    →    통합 플랫폼
수기 입력            (약 1개월)             (10~12개월)
                     HLS 전용              다지점 확장 가능
                     무료 운영             앱 + 분석 기능
```

# 2. 현재 상황 분석

## 2.1 현행 업무 방식의 문제점

현재 HLS 학원에서는 학생 관리, 출결 체크, 상담 기록 등을 **스프레드시트 수기 입력** 방식으로 운영하고 있습니다.

### 업무별 소요 시간 (추정)

| 업무 | 담당자 | 일일 소요 시간 | 월간 환산 |
|------|--------|---------------|----------|
| 출결 체크 및 기록 | 관리자/강사 | 30~60분 | 10~20시간 |
| 학부모 문자 발송 | 관리자 | 20~40분 | 7~13시간 |
| 상담 기록 정리 | 관리자/강사 | 15~30분 | 5~10시간 |
| 학생 정보 검색 | 전 직원 | 10~20분 | 3~7시간 |
| 월말 통계 작성 | 팀장/원장 | - | 3~5시간 |
| **합계** | | **1~2.5시간/일** | **28~55시간/월** |

### 수기 관리의 구조적 한계

| 문제 | 영향 | 빈도 |
|------|------|------|
| **데이터 입력 오류** | 학생 정보 불일치, 출결 누락 | 주 1~2회 |
| **정보 검색 지연** | 상담 시 과거 기록 확인 어려움 | 매일 |
| **동시 작업 충돌** | 여러 명이 수정 시 데이터 덮어쓰기 | 주 1회 |
| **통계 작성 부담** | 월말 수작업 집계로 야근 발생 | 월 1회 |
| **데이터 손실 위험** | 실수로 인한 삭제, 파일 손상 | 연 1~2회 |
| **분점 확장 어려움** | 지점별 데이터 통합 관리 불가 | 상시 |

## 2.2 시스템 도입 시 기대 효과

| 현재 방식 | 시스템 도입 후 |
|----------|---------------|
| 매번 수기 입력 | 클릭 몇 번으로 자동 기록 |
| 정보 찾기 어려움 | 검색으로 즉시 조회 |
| 실수로 데이터 손실 가능 | 자동 백업 및 복구 |
| 여러 명 동시 작업 시 충돌 | 실시간 동시 접속 가능 |
| 통계/분석 수작업 | 자동 리포트 생성 |
| 학부모 문자 일일이 발송 | 자동 알림 발송 |

# 3. Part 1: 스마트 학원 관리 시작하기 (GAS)

## 3.1 개요

Part 1은 **Google Apps Script(GAS)** 기반의 웹 시스템입니다. 브라우저에서 접속하여 사용하며, Google 인프라를 활용하여 **추가 운영 비용 없이** 운영할 수 있습니다.

> **Google Apps Script(GAS)란?**
> Google이 제공하는 클라우드 기반 개발 환경으로, Google 스프레드시트, 드라이브 등과 연동되는 웹 애플리케이션을 만들 수 있습니다. 별도 서버 비용 없이 운영 가능합니다.

### 기술 구성

| 항목 | 기술 | 비용 |
|------|------|------|
| 프론트엔드 | HTML/CSS/JavaScript | 무료 |
| 백엔드 | Google Apps Script | 무료 |
| 데이터베이스 | Google 스프레드시트 | 무료 |
| 호스팅 | Google Cloud | 무료 |
| 문자 발송 | 솔라피 API 연동 | 건당 과금 |

## 3.2 주요 기능

### 3.2.1 학생 관리

| 기능 | 설명 |
|------|------|
| 학생 등록 | 이름, 연락처, 학교, 학년, 학부모 정보 등록 |
| 정보 수정 | 등록된 학생 정보 수정 및 업데이트 |
| 학생 검색 | 이름, 반, 학년 등 조건별 즉시 검색 |
| 수강 이력 | 수강 과목 및 반 변경 이력 자동 기록 |
| 상태 관리 | 재원/휴원/퇴원 상태 관리 |

|------|------|

| 정원 관리 | 반별 정원 설정 및 현황 확인 |

# 4. Part 2: 통합 디지털 학원 시스템 (풀스택)

## 4.1 왜 Part 2가 필요한가

| 요구사항 | Part 1 | Part 2 |
|---------|--------|--------|
| "분점에도 같은 시스템을 쓰고 싶다" | 불가 | 가능 |
| "학생들이 앱으로 문제를 풀었으면 좋겠다" | 불가 | 가능 |
| "문제 은행을 체계적으로 관리하고 싶다" | 불가 | 가능 |
| "학생별 취약점을 자동 분석해주면 좋겠다" | 불가 | 가능 |
| "학생 500명 이상도 빠르게 처리되어야 한다" | 불가 | 가능 |

이러한 기능들은 Google Apps Script의 구조적 한계로 인해 구현이 어렵거나 불가능합니다. **본격적인 웹/앱 서비스로 새로 개발**해야 합니다.

## 4.2 시스템 구성

Part 2는 **3개의 연동된 서비스**로 구성됩니다.

```
┌─────────────────────────────────────────────────────────────────┐
│                      Supabase (통합 백엔드)                       │
│          데이터베이스 + 사용자 인증 + 실시간 동기화 + 파일 저장        │
└─────────────────────────────────────────────────────────────────┘
              │                      │                      │
        ┌─────┴─────┐          ┌─────┴─────┐          ┌─────┴─────┐
        │ 서비스 1   │          │ 서비스 2   │          │ 서비스 3   │
        │           │    ←→    │           │    ←→    │           │
        │ 학생 관리  │          │ 문제 풀이  │          │ 문제 DB   │
        │   웹      │          │   앱      │          │   웹      │
        │ (관리자용) │          │  (학생용)  │          │ (강사용)  │
        │           │          │           │          │           │
        │  React    │          │  Flutter  │          │  React    │
        └───────────┘          └───────────┘          └───────────┘
```

> **Supabase란?**
> 오픈소스 기반 백엔드 서비스로, 데이터베이스, 사용자 인증, 실시간 동기화 등을 제공합니다. 직접 서버를 구축하는 것보다 빠르고 비용 효율적입니다.

> **React란?**
> 페이스북이 개발한 웹 애플리케이션 프레임워크로, 현재 가장 널리 사용되는 웹 개발 기술입니다.

> **Flutter란?**
> Google이 개발한 모바일 앱 개발 프레임워크로, 하나의 코드로 Android와 iOS 앱을 동시에 만들 수 있습니다.

## 4.6 3개 서비스 연동 흐름

세 서비스는 **실시간으로 연동**되어 데이터가 자동으로 흐릅니다.

```
[강사: 문제 DB]                    [학생: 앱]                    [관리자: 학생 관리]
      │                                │                               │
      │  (1) 문제 등록/수정              │                               │
      ├──────────────────────────────→ │                               │
      │     (실시간 반영)               │                               │
      │                                │  (2) 문제 풀이                  │
      │                                ├──────────────────────────────→│
      │                                │     (결과 자동 기록)            │
      │                                │                               │
      │  (3) 정답률/오답 통계             │                               │
      │←──────────────────────────────┤                               │
      │     (분석 데이터)               │                               │
      │                                │                               │
      │                                │  (4) 취약점 분석 결과            │
      │                                │←──────────────────────────────┤
      │                                │     (학습 가이드)               │
```

### 연동 시나리오 예시

| 시나리오 | 흐름 |
|---------|------|
| 새 문제 출제 | 강사가 문제 등록 → 학생 앱에 즉시 반영 |
| 학생 풀이 | 학생이 문제 풀이 → 정오표 자동 생성 → 학습일지에 기록 |
| 취약점 파악 | 오답 누적 → 취약 유형 분석 → 학생관리 시스템에 표시 |
| 학부모 알림 | 주간 성적 집계 → 자동 문자/카톡 발송 |

## 4.7 개발 범위에서 제외되는 것

다음 기능들은 기술적 복잡도, 규정 준수 요건 등으로 인해 **본 프로젝트 범위에 포함되지 않습니다**.

| 제외 항목 | 이유 | 대안 |
|----------|------|------|
| **결제 시스템** | PG연동, PCI-DSS 보안 규정 준수 필요 | 토스페이먼츠 등 전문 솔루션 도입 |
| **키오스크** | 하드웨어 연동 + 결제 기능 필요 | 별도 전문 업체 |
| **iOS 앱** | 개발 기간/비용 증가 | Android 우선, iOS는 추후 협의 |
| **오프라인 모드** | 동기화 복잡도 높음 | 필요시 추후 추가 개발 |
| **주관식 자동 채점** | AI 기술 필요, 정확도 보장 어려움 | 객관식/단답형 우선 |

> **특히 결제 시스템**은 금융 보안 규정이 엄격하여, 무리하게 포함하면 프로젝트 전체 실패 위험이 높아집니다. 전문 결제 솔루션(토스페이먼츠, 네이버페이, 카카오페이 등) 도입을 권장합니다.

# 7. 소유권 및 권리 구조

## 7.1 소프트웨어 권리의 기본 개념

소프트웨어 개발에서 **"누가 무엇을 소유하는가"**는 중요한 문제입니다. 흔히 혼동되는 개념을 명확히 구분합니다.

```
┌────────────────────────────────────────────────────────────────────┐
│                    소프트웨어 권리 3가지 개념                          │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│  (1) 소스코드 인도 ───→  "집 열쇠를 받는 것"                         │
│      코드 파일/저장소 접근권을 받는 것                                 │
│                                                                    │
│  (2) 저작재산권 양도 ──→  "집 등기를 이전받는 것"                     │
│      복제/배포/수정 등 경제적 권리를 넘기는 것                         │
│                                                                    │
│  (3) 사용권 부여 ────→  "임차권을 받는 것"                           │
│      특정 범위 내에서 사용할 수 있는 권리                              │
│                                                                    │
│  ※ 소스코드를 받았다고 자동으로 저작권이 넘어가지 않습니다.             │
│     계약서에 별도로 명시해야 합니다.                                   │
│                                                                    │
└────────────────────────────────────────────────────────────────────┘
```

## 7.2 저작권 귀속의 법적 원칙

| 상황 | 기본 원칙 | 실무 해석 |
|------|----------|-----------|
| **외주/위탁 개발** | 계약으로 정함 | 계약서에 IP 조항 명시 필수 |
| **정규직(업무상저작물)** | 법인 등이 저작자 | 회사 귀속으로 해석될 가능성 높음 |
| **저작재산권 양도 계약** | 계약서 문구가 핵심 | 양도 범위 미기재 시 분쟁 가능 |

> **법적 근거**: [저작권법 제9조(업무상저작물) 해설 - 문화체육관광부](https://www.mcst.go.kr/kor/s_policy/dept/deptView.jsp?pSeq=1305&pDataCD=0417000000&pType=01), [저작권법 제45조(저작재산권의 양도) - 법령정보센터](https://www.law.go.kr/lsLinkProc.do?lsNm=%EC%A0%80%EC%9E%91%EA%B6%8C%EB%B2%95&ancYnChk=0&mode=20&0&chrClsCd=010202&joNo=004500000&efYd=20251223#)

## 7.3 분점/프랜차이즈 확장을 위해 필요한 권리

만약 이 시스템을 **분점이나 가맹점에도 배포**하려면, 아래 권리들이 필요합니다:

| 권리 | 왜 필요한가 |
|------|------------|
| **복제권** | 지점이 늘어날 때마다 시스템을 설치/복사해야 함 |
| **개작(수정)권** | 지점별로 기능을 추가/변경할 수 있어야 함 |
| **서브라이선스권** | 가맹점에게 "사용권"을 내려줄 수 있어야 함 |

> **"본점에서만 쓰는 내부 시스템"**과 **"프랜차이즈 확장 가능한 시스템"**은 계약상 완전히 다른 상품입니다. 후자가 더 넓은 권리를 포함하므로 비용도 달라집니다.

## 7.4 권리 구조 옵션

학원의 필요에 따라 두 가지 옵션을 제안합니다.

### 옵션 A: 기본형 (월 700만원)

| 항목 | 내용 |
|------|------|
| **소스코드** | 전체 인도 (저장소 접근권 부여) |
| **사용권** | HLS 학원 내 무제한 사용 |
| **수정권** | 학원 내부에서 자유롭게 수정 가능 |
| **유지보수** | 다른 개발자 투입 가능 |
| **저작권** | 개발자 보유 |

**학원이 할 수 있는 것**:
- HLS 모든 지점에서 시스템 사용
- 필요에 따라 기능 수정/추가
- 다른 개발자를 투입하여 유지보수
- 시스템을 기반으로 사업 운영

**학원이 할 수 없는 것**:
- 소스코드를 외부에 판매
- 타 학원(비계열)에 시스템 제공
- 시스템 자체를 상품으로 판매

### 옵션 B: 완전 양도형 (월 800만원)

| 항목 | 내용 |
|------|------|
| **소스코드** | 전체 인도 (저장소 접근권 부여) |
| **저작재산권** | **전체 양도** (복제, 배포, 수정, 2차 저작물 작성 등) |
| **사용 범위** | 제한 없음 |

**학원이 할 수 있는 것**:
- 옵션 A의 모든 권리
- 소스코드를 외부에 판매
- 타 학원에 시스템 제공/판매
- 시스템 자체를 상품으로 사업화
- 개발자 동의 없이 자유로운 활용

## 7.6 왜 저작권 양도에 프리미엄이 있는가

저작권 완전 양도는 개발자 입장에서 다음을 의미합니다:

| 포기하는 것 | 설명 |
|------------|------|
| 재사용 권리 | 동일 코드를 다른 프로젝트에 활용 불가 |
| 포트폴리오 | 향후 유사 프로젝트 수주 시 활용 제한 |
| 파생 수익 | 시스템 상품화 시 수익 참여 불가 |

따라서 완전 양도 시에는 업계 관행상 **개발비의 10~30% 프리미엄**이 적용됩니다. 본 제안에서는 **월 100만원 (약 14%)** 프리미엄을 반영하였습니다.

## 7.7 옵션 전환 가능 (크레딧 상계 방식)

옵션 A로 시작하다가 추후 저작권 완전 양도(옵션 B)가 필요해지면 별도 협의를 통해 전환할 수 있습니다. 이 경우 **기존 지급액의 일부를 크레딧으로 인정**하는 방식도 협의 가능합니다.

예시:
```
- 12개월간 옵션 A (월 700만원) 지급 = 8,400만원
- 나중에 완전 양도 협의 시, 기존 지급액의 일부를 양도가에서 차감
→ "이중 지불" 부담을 줄이는 업계 표준 방식
```

## 7.8 권장 사항

| 상황 | 권장 옵션 |
|------|----------|
| HLS 계열 지점에서만 사용 예정 | **옵션 A** (비용 효율적) |
| 타 학원 판매/프랜차이즈 사업 계획 | **옵션 B** (완전한 권리 확보) |
| 현재는 내부 사용, 추후 확장 가능성 | **옵션 A** → 나중에 옵션 B로 전환 협의 |

## 7.9 정규직 전환 시 IP 구조

정규직으로 전환하는 경우, IP 귀속 구조가 달라집니다.

| 비교 축 | 외주 (옵션 A) | 외주 (옵션 B) | 정규직 |
|---------|-------------|-------------|--------|
| 저작권 귀속 | 개발자 | 학원 | 회사 귀속 가능성 높음 |
| 월 보상 | 700만원 | 800만원 | 별도 협의 |
| 고용 안정성 | 낮음 | 낮음 | 높음 (근로기준법 보호) |
| 4대보험 | 본인 부담 | 본인 부담 | 회사 부담 |
| 퇴직금 | 없음 | 없음 | 발생 |

> **정규직 전환은 외주 옵션과 별도로 협의**가 필요합니다. 정규직의 경우 고용 안정성과 복리후생이 있는 대신, 업무상저작물로 IP가 회사에 귀속될 수 있어 **월 보상 수준이 달라져야** 합니다.
>
> **법적 근거**: [근로기준법(해고 제한)](https://www.law.go.kr/LSW//lsInfoP.do?lsiSeq=265959&efYd=20251023#0000), [근로자퇴직급여 보장법](https://www.law.go.kr/lsLinkProc.do?lsNm=%EA%B7%BC%EB%A1%9C%EC%9E%90%ED%87%B4%EC%A7%81%EA%B8%89%EC%97%AC%20%EB%B3%B4%EC%9E%A5%EB%B2%95&chrClsCd=010202&mode=20&ancYnChk=0&lsId=000746&chrCls=010202&joNo=000800000&viewCls=lsRvsDocInfoR)

# 10. 자주 묻는 질문

## Q1. Part 1만 하고 Part 2는 안 해도 되나요?

**네, 가능합니다.**

Part 1만으로도 현재 스프레드시트 수기 입력 방식보다 **훨씬 효율적**입니다. Part 1을 먼저 사용해보시고, 다음과 같은 필요가 생길 때 Part 2를 진행하시면 됩니다:

- 학생 수가 500명 이상으로 증가
- 분점 확장 계획
- 학생용 앱 필요
- 고급 분석 기능 필요

## Q2. 개발 완료 후 개발자가 그만두면요?

걱정하지 않으셔도 됩니다:

| 대비책 | 설명 |
|--------|------|
| 소스코드 인도 | 모든 코드가 학원에 인도되므로 다른 개발자가 유지보수 가능 |
| 표준 기술 사용 | React, Flutter 등 업계 표준 기술로 개발하여 인수인계 용이 |
| 문서화 | 시스템 구조 및 사용법 문서 제공 |
| 저장소 접근권 | GitHub 등 코드 저장소 접근권 이전 |

## Q3. 나중에 옵션 A에서 옵션 B로 바꿀 수 있나요?

**네, 가능합니다.**

옵션 A로 진행하다가 추후 저작권 완전 양도(옵션 B)가 필요해지면 별도 협의를 통해 전환할 수 있습니다. 기존 지급액의 일부를 크레딧으로 인정하는 방식도 협의 가능합니다.

## Q4. 왜 결제 시스템은 안 만드나요?

결제 시스템은 다음과 같은 이유로 별도 전문 솔루션을 권장합니다:

| 이유 | 설명 |
|------|------|
| **보안 규정** | PCI-DSS 등 금융 보안 규정 준수 필요 |
| **PG사 계약** | 결제대행사(PG)와 별도 계약 필요 |
| **책임 소재** | 결제 오류 시 금전적 피해 직결 |
| **전문성** | 결제 분야는 별도의 전문 지식 필요 |

**권장 대안**: 토스페이먼츠, 네이버페이, 카카오페이 등 전문 결제 솔루션 도입

## Q5. iOS 앱은 왜 안 만드나요?

초기 버전에서는 Android만 개발하는 이유:

| 이유 | 설명 |
|------|------|
| **개발 비용** | iOS 추가 시 개발 비용 30~50% 증가 |
| **출시 복잡도** | Apple 앱스토어 심사가 까다로움 |
| **시장 점유율** | 한국 모바일 시장에서 Android 점유율이 높음 |
| **학원 환경** | 학원 비치 태블릿은 대부분 Android |

iOS 앱은 Android 앱 안정화 후 **추후 협의**를 통해 추가 개발 가능합니다.

## Q6. Part 1 데이터를 Part 2로 옮길 수 있나요?

**네, 가능합니다.**

Part 1(GAS)에서 축적된 학생, 반, 출결, 상담 데이터는 Part 2(풀스택) 개발 시 **마이그레이션(이전)** 작업을 통해 그대로 옮길 수 있습니다. 데이터 손실 없이 연속적으로 사용 가능합니다.

## Q7. 개발 중에 요구사항이 바뀌면요?

**백로그 방식**으로 유연하게 대응합니다:

| 기존 방식 | 백로그 방식 |
|----------|------------|
| 처음에 전체 범위 확정 | 매월 "이번 달 할 일" 합의 |
| 범위 변경 시 재협상 필요 | 우선순위 조정이 자연스러움 |
| 요구사항 변경에 취약 | 변화에 유연하게 대응 |

> **백로그(Backlog)**: 개발할 기능들의 우선순위 목록. 매월/매분기 검토하여 순서를 조정합니다.

## Q8. 옵션 A와 옵션 B, 어떤 걸 선택해야 하나요?

| 판단 기준 | 옵션 A 선택 | 옵션 B 선택 |
|----------|------------|------------|
| 사용 목적 | HLS 내부 사용 중심 | 외부 판매/사업화 계획 |
| 분점 배포 | HLS 계열 지점만 | 비계열 타 학원도 포함 |
| 비용 우선 | 월 100만원 절약 | 완전한 권리 확보 우선 |
| 향후 계획 | 당장은 내부 사용 | SaaS 사업화 고려 중 |

**대부분의 학원에는 옵션 A가 적합합니다.** 외부 판매/사업화 계획이 있을 때 옵션 B를 선택하시면 됩니다.
